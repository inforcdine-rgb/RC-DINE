1. ABSTRACT

R-C DINE is a web-based restaurant and hotel order management system designed to digitize dine-in operations. The system supports two primary user roles—Owner and Manager—and a public Customer interface that does not require login.

Owners can register multiple hotels (up to ten per account), invite managers via email, assign managers to hotels, and manage subscriptions. Managers operate day-to-day functions: menu management, table management with QR-code-based ordering, live order tracking, payment confirmation, and analytics dashboards. Customers scan a table QR code, register basic details, browse the menu, place orders, pay online or request manual payment approval, and submit feedback.

The application follows a three-tier architecture: React frontend (presentation), Node.js/Express backend (business logic and REST APIs), and MySQL database (persistent storage). JWT secures authenticated routes; Razorpay handles subscriptions and customer payments; Web Push and service workers support real-time notifications.

Development-mode enhancements allow local testing without email verification. The project demonstrates practical skills in full-stack development, role-based access control, payment gateway integration, and real-world workflow automation suitable for diploma-level evaluation.

Keywords: Restaurant Management, QR Ordering, React, Node.js, Express, MySQL, JWT, Razorpay, Role-Based Access Control.

2. INTRODUCTION

The hospitality industry increasingly depends on digital tools to reduce manual errors, speed up service, and improve customer experience. Traditional paper menus and verbal order-taking are slow, error-prone, and difficult to track.

R-C DINE (Restaurant Operations System) addresses these challenges by providing a unified platform where:

  • Hotel owners manage multiple properties from one account.
  • Managers run operations for a selected hotel.
  • Customers order food directly from their smartphones using QR codes placed on tables.

The project is implemented as a Single Page Application (SPA) using React on the client and a RESTful API server using Node.js and Express. Data is stored in a relational MySQL database accessed through Sequelize ORM.

This report documents the complete system—from problem definition and design to implementation, testing, limitations, and future improvements—in a format suitable for diploma submission and viva voce examination.

3. PROBLEM STATEMENT

In many small and medium restaurants:

  1. Manual ordering leads to miscommunication between customer, waiter, and kitchen.
  2. No centralized record of orders, revenue, or table occupancy exists.
  3. Multiple hotels under one owner are difficult to monitor without separate tools.
  4. Staff access control is weak—everyone cannot safely share one login.
  5. Payment tracking is inconsistent between cash, UPI, and card.
  6. Customer experience lacks modern contactless ordering options (especially post-COVID expectations).

There is a need for an affordable, customizable, web-based system that supports multi-hotel management, role-based staff access, QR-based customer ordering, and online payment integration.

15. AUTHENTICATION & LOGIN FLOW

15.1 Registration

  1. User fills signup form (Formik + Yup).
  2. Password encrypted with AES (crypto-js) using CRYPTO_SECRET_KEY.
  3. POST /api/user/register with encrypted password.
  4. Backend decrypts → validates → saves user + preferences.
  5. Production: status INACTIVE, verification email sent.
  6. Development: status ACTIVE, email skipped (current project fix).

15.2 Login

  1. Password encrypted on client.
  2. POST /api/user/login.
  3. Backend finds user, decrypts stored password, compares.
  4. Checks verification status (skipped in dev).
  5. For managers: checks hotel subscription not expired.
  6. Returns JWT token + encrypted role/hotel data.
  7. Stored in localStorage; axios adds Authorization header.

15.3 Session Protection

  • No token → Auth routes redirect to /.
  • Invalid token → API returns error → localStorage cleared → redirect login.
  • Owner/Manager sidebars show different menu tabs.

36. VIVA QUESTIONS & ANSWERS

Q1. What is R-C DINE?
A: R-C DINE is a web-based restaurant/hotel management system for owners, managers, and customers using QR-based ordering.

Q2. Which architecture is used?
A: Three-tier: React (presentation), Node/Express (application), MySQL (data).

Q3. What is JWT?
A: JSON Web Token—a signed string stored after login used to authenticate API requests.

Q4. Why encrypt password on frontend?
A: To avoid sending plain text over HTTP; backend decrypts with shared secret before hashing/storage.

Q5. Difference between Owner and Manager?
A: Owner manages hotels, invites, subscriptions. Manager runs menu, tables, orders for assigned hotel.

Q6. How does QR ordering work?
A: Table ID is encrypted into URL; customer opens /place/token, registers, and orders without login.

Q7. Which database tables store orders?
A: orders table linked to customers and menus.

Q8. What is Sequelize?
A: ORM that maps JavaScript objects to MySQL tables.

Q9. What is Redux Saga?
A: Middleware that handles async operations (API calls) in response to Redux actions.

Q10. How is Razorpay integrated?
A: Backend creates Razorpay orders/subscriptions; frontend opens checkout; confirm endpoint verifies payment.

Q11. What happens when Add Hotel is clicked?
A: Modal opens → form submit → POST /api/hotel → hotel + owner relation saved in MySQL.

Q12. What is CORS?
A: Cross-Origin Resource Sharing—browser security; backend allows frontend origin to call APIs.

Q13. What are order statuses?
A: PENDING, SERVED, CANCELLED.

Q14. Why MySQL over MongoDB here?
A: Relational data (users, hotels, orders) fits tables and foreign keys naturally.

Q15. What was the signup bug you fixed?
A: User saved to DB but email failed, causing duplicate error on retry; fixed with transactions and dev-mode skip.
